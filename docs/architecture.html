UI layer

fragments hosted in the MainActivity drawer shell · renders cached data only

ui.splash.SplashActivitylauncher

core-splashscreen with a custom exit animation, then an explicit intent (with extra) into MainActivity.

MainActivity

MaterialToolbar + NavigationDrawer + fragment container. Home is the backstack root; other screens push with custom slide/fade transitions. Owns the refresh action, the connectivity check and the POST_NOTIFICATIONS runtime-permission flow.

ui.homeViewPager2

HomeFragment pages one dynamically-created CityWeatherPageFragment per saved city; selected page survives rotation. Empty state links to Cities. Share / open-in-maps implicit intents.

ui.forecastRecyclerView

ForecastAdapter (ListAdapter + DiffUtil, custom ViewHolder) renders 7-day CardView rows for the home city.

ui.citiesContentResolver

Reads and mutates cities exclusively through the ContentProvider, with a ContentObserver for live updates. Add = geocoding search dialog (manual lat/lon fallback); long-press = edit/delete dialogs; star = home city.

ui.settings

PreferenceFragmentCompat over the shared pref_key_* resources: units, sync interval, notifications toggle.

ui.about

Version info and an implicit browser intent to the data source.

ui.UnitsFormatter · ui.anim

Display-time °C/km/h → °F/mph conversion (storage stays metric) and the reusable animation helpers (fadeInUp, staggerIn, …).

WeatherDataSource

Data layer

SQLite + prefs, one shared DB helper

data.db.WeatherDbHelperSQLiteOpenHelper

Tables cities, current_weather (1:1, upsert), daily_forecast (replaced per refresh); FK cascade on city delete.

data.repository

SqliteCityRepository and SqliteWeatherRepository singletons; refresh persists transactionally; mutations notify both listeners and the provider URI.

data.provider.CityProviderContentProvider

Authority hr.alg.iamu_project_bp.provider, dir/item URIs + MIME types over the cities table. Not exported; consumed by the Cities screen via ContentResolver.

data.prefs.Preferences.kt

Extension-function access to the static default file (units, interval, notifications, home city) and dynamic per-city files.

Background processing

WorkManager · AlarmManager · broadcasts · notifications

work.WeatherSyncWorkerCoroutineWorker

Refreshes all cities (or one via input data) through the repositories; retries on IOException; posts a sync notification.

work.SyncScheduler

Unique periodic work from the interval preference (CONNECTED constraint) + one-shot sync used when a city is added or edited.

alarm.AlarmScheduler

Exact weather-alert alarms (setExactAndAllowWhileIdle, windowed fallback) with broadcast PendingIntents.

receiver

WeatherAlertReceiver turns alarm broadcasts into alert notifications; BootReceiver re-schedules sync after reboot.

notification.NotificationHelper

Creates both channels; gates posting on the runtime permission and the user preference; content PendingIntents open MainActivity.

notification.DelayedExecutorHandler/Looper

HandlerThread + postDelayed utility; auto-dismisses the sync notification.

Data flow — manual refresh

  1. MainActivity refresh action → ConnectivityChecker (offline → Snackbar, stop)
  2. AppGraph.weatherRepository().refresh(homeCityId) in lifecycleScope
  3. OpenMeteoWeatherDataSource.fetchWeather(lat, lon) → Retrofit on IO → WeatherResponseParser
  4. SqliteWeatherRepository upserts current_weather + replaces daily_forecast in one transaction
  5. Snackbar feedback; fragments re-read the cache on next render (onResume re-renders apply units changes too)

Data flow — adding a city

  1. Cities FAB → search dialog → CitySearch.searchCities(name) → pick "Name — Country"
  2. ContentResolver.insert → CityProvider → shared DB → notifyChange (ContentObserver refreshes the list)
  3. SyncScheduler.triggerOneShotSync(cityId) → WeatherSyncWorker fetches weather immediately
